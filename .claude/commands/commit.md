Create a commit for the currently staged changes. Follow these steps:

1. Run `git status` to see what files are staged
2. Run `git diff --staged` to see the actual changes that will be committed
3. Analyze the changes and create an appropriate commit message that:
   - Summarizes the nature of the changes
   - Uses the format: `{component} {emoji}: {message}` — e.g. `nvim ♻️: Refactoring LSP config` or `claude ✨: Adding new skill`
   - The component is the name of the changed area (e.g. `nvim`, `claude`, `bin`, `hypr`) — never a type like `feat` or `fix`
   - The emoji indicates the type of change (see table below)
   - Is concise but descriptive
   - Never add comments like `Generated by Claude` or `Generated by Claude AI`
   - Using the writing-clearly-and-concisely skill to make the commit message more clear for later reading
4. Show the generated message to the user in a code block format and say: "Here's the proposed commit message. Do you want to continue?"
5. If the user wants to continue, run `git commit`
6. Confirm the commit was successful with `git status`

If there are no staged changes, inform the user that there's nothing to commit.

 # Git Commit : Well Formatted Commits

## Best Practices for Commits

- **Verify before committing**: Ensure code is linted, builds correctly, and documentation is updated. Make sure to run ALL of the pre-commit checks first.
- Commit messages should use present participle (gerund) form, e.g. "Adding feature" instead of "Add feature", "Fixing bug" instead of "Fix bug", "Removing unused code" instead of "Remove unused code"
- **Concise first line**: Keep the first line under 72 characters
- **Emoji**: Each commit type is paired with an appropriate emoji:
  - ✨ `feat`: New feature
  - 🐛 `fix`: Bug fix
  - 📝 `docs`: Documentation
  - 💄 `style`: Formatting/style
  - ♻️ `refactor`: Code refactoring
  - ⚡️ `perf`: Performance improvements
  - ✅ `test`: Tests
  - 🔧 `chore`: Tooling, configuration
  - 🚀 `ci`: CI/CD improvements
  - 🗑️ `revert`: Reverting changes
  - 🧪 `test`: Add a failing test
  - 🚨 `fix`: Fix compiler/linter warnings
  - 🔒️ `fix`: Fix security issues
  - 👥 `chore`: Add or update contributors
  - 🚚 `refactor`: Move or rename resources
  - 🏗️ `refactor`: Make architectural changes
  - 🔀 `chore`: Merge branches
  - 📦️ `chore`: Add or update compiled files or packages
  - ➕ `chore`: Add a dependency
  - ➖ `chore`: Remove a dependency
  - 🌱 `chore`: Add or update seed files
  - 🧑‍💻 `chore`: Improve developer experience
  - 🧵 `feat`: Add or update code related to multithreading or concurrency

# Using Jira ticket number

In case the Jira ticket number was given as argument, the commit message will be prefixed with the ticket number.
If the ticket number is not given, check the current branch name, if you can get the ticket number can be taken from it.
The ticket number always starts with `VERBU-` followed bu 5 numbers.
If not, it might not be a Jira related commit, so just create the commit message without the ticket number.
