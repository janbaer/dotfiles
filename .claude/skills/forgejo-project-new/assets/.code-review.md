# Code Review Guidelines

## ❌ Absolute Prohibitions — Read This First

These rules override everything else. Violating any of them makes the review wrong, regardless of its technical content:

- **NO summary.** Do not describe what the PR does, what changed, or how it works.
- **NO compliments.** Do not acknowledge good code, clean structure, minimal diffs, or well-scoped changes.
- **NO "Issue Alignment", "Code Quality", "Suggestions", or any other invented section.** Use only the sections defined in the Output Format below.
- **NO filler.** Do not write "No naming concerns", "No security issues found here", or any other statement about categories that have no findings.
- **If there are no issues at all:** output exactly one line — `No issues found.` — then the Assessment. Nothing else.

## What to Check

**Security:** SQL injection, XSS, auth/authz gaps, secrets in code, path traversal, CSRF, unsafe input, insecure crypto

**Correctness:** Logic errors, off-by-one, race conditions, null dereferences, unhandled edge cases

**Performance:** N+1 queries, missing indexes, memory leaks, blocking ops, excessive network calls

**Code quality:** DRY violations, unclear naming, missing error handling, hard-coded values, tight coupling, functions >50 lines

**Best practices:** Language idioms, framework conventions, type safety, null handling, test coverage

**Requirements:** All plan requirements met? Implementation matches spec? No scope creep? Breaking changes documented?

## Output Format

Only use this exact structure. Do not add, rename, or reorder sections.

### 🔴 Critical / 🟡 Medium / 🟢 Low — Short title (file:line)

What is wrong and why it matters. What needs to change.

One entry per issue. Group by severity (🔴 first). One paragraph per issue — no more.

If there are no issues: `No issues found.`

Then always end with:

### Assessment

**Ready to merge?** [Yes / No / With fixes]
**Reasoning:** [One or two sentences — technical only, no praise.]
